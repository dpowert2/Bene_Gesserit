# Architecture

## Goals

1. **Single-tenant fork-and-deploy.** Every user runs their own deployment with their own data and their own thesis. No multi-tenant infrastructure to operate.
2. **Few external services.** GitHub for code, Vercel for runtime, Turso for state, Anthropic for AI. That's it.
3. **Right architecture over minimum services.** State goes in a real database, not in commit history. The DB layer is portable across libSQL/Postgres via Drizzle.
4. **Personalization is data, not code.** Thesis, scoring weights, search terms, vertical-signal naming all live in the database and are edited via the UI. The repo ships sane defaults; users never edit code to use BG.
5. **Agents are first-class.** The hard part of BG is the AI work. Agent code is in `/agents`, plain TypeScript, version-controlled, fork-friendly.

## Data Layer

Drizzle ORM + libSQL driver. Schema in `db/schema.ts`. Migrations are SQL files in `db/migrations/`, generated by `drizzle-kit generate` and applied by `drizzle-kit migrate` on deploy.

Tables:

- **`startups`** — every candidate, scored. Fields: slug (PK), name, url, oneliner, gap, stage_label, geo, vertical_signal, team, thesis_fit, evidence (JSON), score, bars (JSON), source ('radar' | 'manual' | 'imported'), manual_add (bool), detected_at, created_at.
- **`votes`** — per-card user state. Fields: slug (PK, FK → startups), voter, stars, note (history log), followup, archived, kanban_stage, voted_at.
- **`manual_pending`** — URLs submitted via the kanban awaiting agent processing. Fields: slug (PK), url, name, status, error_msg, submitted_at, resolved_at.
- **`radar_runs`** — log of scheduled scans. Fields: id (PK), run_date, status, new_slugs (JSON), summary, created_at, completed_at.
- **`changelog_entries`** — every system change worth recording. Fields: id (PK), type, date, title, body, evidence, source, created_at.
- **`config`** — key/value store for thesis (markdown), scoring_weights (JSON), search_terms (markdown), voter_name, vertical_signal_label, etc. Fields: key (PK), value, updated_at.

Why libSQL not Postgres: BG's workload is small (single user, low write rate, modest data volume). SQLite semantics are sufficient and simpler. Edge replication via Turso gives global low-latency reads. Drizzle's libSQL adapter and Postgres adapter are interchangeable enough that swapping requires only updating the connection module — anyone who needs Postgres can do it.

## Agent Layer

The agent code lives in `/agents/`. Each agent is a single TypeScript module exporting an async function. The functions are invoked by Vercel API routes (for on-demand) or Vercel Cron (for scheduled).

Pattern (simplified):

```ts
// agents/generate-card.ts
import Anthropic from "@anthropic-ai/sdk";
import { db } from "@/db/client";
import { startups, manual_pending, config } from "@/db/schema";

export async function generateCard(url: string, slug: string) {
  const thesis = await db.select().from(config).where(eq(config.key, "thesis"));
  const scoringWeights = await db.select().from(config).where(eq(config.key, "scoring_weights"));
  const verticalLabel = await db.select().from(config).where(eq(config.key, "vertical_signal_label"));

  const client = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

  // Multi-turn agentic loop: research → score → format
  // Uses web_search tool for sourcing, structured output for the card.
  const card = await client.messages.create({ /* ... */ });

  await db.insert(startups).values({
    slug, name: card.name, url, oneliner: card.oneliner,
    /* ... */ source: "manual", manual_add: true,
    detected_at: new Date().toISOString().slice(0, 10),
    created_at: new Date().toISOString(),
  });
  await db.update(manual_pending).set({ status: "done", resolved_at: ... }).where(eq(manual_pending.slug, slug));
}
```

The full agent is naturally longer (research loop, scoring, JSON-mode prompting), but the shape is "stateless function that reads config + writes results". No special infrastructure beyond the Vercel function runtime.

## Auth

Single-user. An environment variable `BG_PASSWORD` defines the password. A middleware (`middleware.ts`) checks for a signed cookie on every request; the cookie is set by `POST /api/login` after the password matches. Cookie lifetime is 30 days; `BG_PASSWORD` rotation invalidates outstanding cookies because the signing key derives from the password.

This is intentionally minimal. For multi-user deployments, swap in Clerk or NextAuth — the auth touch points are isolated to `middleware.ts`, `app/login/page.tsx`, and `app/api/login/route.ts`.

## Configuration

Three configs are user-defined: thesis, scoring weights, search terms. They live in the `config` table and are edited at `/settings`. On first deploy, the seed script (`db/seed.ts`) reads `/config/*.example` files and inserts them into the DB if no values exist.

This means:

- The repo ships with sensible defaults (every user gets the same starting point).
- The user edits in the UI, not in files (no commit-and-redeploy loop for routine tuning).
- Pulling upstream template changes won't clobber the user's thesis (the .example files don't conflict with DB rows).

## Cron Jobs

Defined in `vercel.json`. Each cron path maps to a Next.js API route under `/app/api/cron/*`.

- `run-radar` — every two days. Reads thesis + search terms from `config`. Runs the radar agent. Inserts new candidates into `startups` and a row into `radar_runs`.
- `process-pending` — every 5 minutes. Selects rows from `manual_pending` where `status = 'pending'`. Calls `generateCard()` for each. Updates the row to `done` or `failed`.
- `changelog-summary` — weekly. Reviews recent radar runs and config changes; writes a summary `changelog_entries` row.

Vercel hobby tier has a 60s function timeout. Single card generation typically completes in 15-40s. Radar runs that exceed 60s fan out: the orchestrator enqueues N pending entries (one per candidate slug to research), and `process-pending` picks them up over subsequent ticks.

## Why Not...

- **GitHub-as-database.** Each vote click would be a commit. No transactions, no indexes, fan-out problems on concurrent writes, repo history pollution. Acceptable for a side project, wrong for a thing you want to use daily.
- **Routines.** Bound to an Anthropic Pro+ plan. A user with just an API key can't run them. Agent SDK in Vercel functions runs anywhere with `ANTHROPIC_API_KEY`.
- **Inngest / Trigger.dev.** Would add another service. Vercel Cron is sufficient; if a user genuinely needs richer workflow orchestration later, they can add it themselves.
- **Cloudflare Pages + Workers.** Tighter platform binding (D1 instead of Turso would be more natural). Vercel + Turso is the lowest-friction default for the Next.js community; Cloudflare is a documented alternative for Cloudflare-native users.
- **Multi-tenant SaaS.** Adds account management, billing, isolation. Out of scope. Each user runs their own deployment.
