# Bene Gesserit — Search Terms
Organised by gap category. Each query is designed for web search.
Rotate through these across scan runs to avoid repetition.
IMPORTANT DESIGN PRINCIPLE: Cast wide on discovery, filter tight on scoring.
Search terms should find the CATEGORY of company, not pre-filter for finserv.
The scoring model (Gap Fit, FinServ Readiness, Stage Match, Team Signal)
does the filtering. If you only search for "compliance fintech", you miss
horizontal platforms that are one conversation away from a finserv vertical.

## SEARCH STRATEGY NOTES

### What changed (April 2026 update)
- DOUBLED the number of search terms per category
- Added "broader discovery" tier to each gap — these use GENERIC
  startup/funding language instead of finserv-specific keywords
- Added Section 7 (source-specific) to mine funding trackers, VC
  portfolios, accelerators, and regional sources directly
- Added Section 8 (competitive intelligence) to find startups in
  the orbits of known players
- Removed finserv keywords from ~60% of searches — the scoring
  model handles finserv filtering, not the search terms

### Why
The previous term set was too narrow. It pre-filtered for finserv in
the search query, which meant horizontal AI compliance/security
startups (the majority of the market) were invisible. Companies like
Alinia, JetStream, Manifold, and Straiker were not being surfaced
because they don't self-describe as "fintech" — they say "enterprise
AI governance" or "agentic security." The finserv relevance signal
belongs in the SCORING layer (FinServ Readiness dimension), not the
SEARCH layer.

### Rotation guidance
With ~80 search terms across 8 sections, each scan run should:
1. Pick 2-3 terms from the ACTIVE gap category's core searches
2. Pick 1-2 terms from the same category's broader discovery
3. Pick 1 term from Section 7 (source-specific)
4. Pick 1 term from Section 8 (competitive intel) — rotate through
   these slowly, one per scan

This gives 5-7 searches per scan run, manageable within token budgets
while covering much more ground than the original 2-3 narrow queries.
